## What I Found

- The repository is a production monorepo for the public site, admin app, shared packages, Supabase migrations, MoPlayer Classic Android, and MoPlayer Pro Android.
- MoPlayer Pro Android lives in `apps/moplayer2-android`. It is a native Kotlin / Jetpack Compose Android and Android TV app, not React Native or Expo.
- The public website lives in `apps/web` and owns activation APIs, release/download routes, public product pages, diagnostics/events, and support.
- The admin app lives in `apps/admin` and already reads devices, licenses, activation requests, runtime config, releases, diagnostics, and support data.
- Shared product slugs live in `packages/shared/src/app-products.ts`. The public Pro name is `MoPlayer Pro`, but API/database product identity remains `moplayer2`.
- Activation is website-backed:
  - Android calls `POST /api/app/activation/create`.
  - The website creates a device row and a short-lived activation code.
  - The website/admin confirms activation.
  - A fresh QR handoff can deliver one encrypted Xtream/M3U source through `GET /api/app/activation/source`.
  - The app acknowledges import through `/api/app/activation/source/ack`.
- Provider credentials must not be permanently stored in Supabase. The app saves the imported or typed source locally, then plays directly from the provider server.
- MoPlayer Pro Android supports M3U/M3U8, Xtream login, QR activation, Home, Live TV, Movies, Series, Episodes, Search, Favorites, Settings, license/status messaging, and player error/fallback states.

## Windows Technology Choice

The Windows app will be added as `apps/moplayer-pro-windows`.

Chosen stack:

- Electron for a real installable desktop app and Windows packaging.
- React + TypeScript + Vite for the renderer.
- Tailwind CSS via the official Vite plugin for fast, maintainable styling.
- `hls.js` for HLS/M3U8 playback on top of the Chromium HTML5 video element.
- `electron-builder` for NSIS installer and portable EXE output.
- Electron `safeStorage` through the main process for local credential encryption where Windows encryption is available.

Reason: the Android Pro app is native Kotlin/Compose and cannot be reused directly on Windows. Electron lets the PC app share the existing web activation APIs and ship as a real Windows desktop application without changing Android or production backend behavior.

## Risks

- A true one-to-one feature port of the Android Kotlin app is a large product effort. This plan ports the functional PC experience and core flows first without changing Android.
- Browser/Chromium playback cannot support every codec that Android Media3/LibVLC can support. HLS/M3U8 is supported through `hls.js`; some MPEG-TS or proprietary streams may need external VLC support later.
- Huge Xtream libraries can be tens of thousands of items. The initial Windows version will cache JSON locally and use UI filtering/pagination. A future upgrade should move the library cache to SQLite if real provider QA shows memory or startup issues.
- Electron installer output is too large for routine Vercel public assets. If the final installer is large, Supabase Storage, GitHub Releases, or Cloudflare R2 should host the binary while the website links to signed or public release metadata.
- The existing worktree is dirty with many unrelated Android/admin/docs changes. Windows work must avoid reverting or overwriting those files.

  - first-run device ID persists;
  - activation creates a real `moplayer2` QR/code with `platform=windows`, `device_type=pc`;
  - Xtream form validates and handles network errors clearly;
  - M3U URL import parses a legal user-provided playlist;
  - Live/Movies/Series/Search/Favorites/Settings are not empty when source data exists;
  - player shows loading, buffering, retry, and failed-source states;
  - fullscreen and shortcuts work;
  - relaunch preserves settings/source/library;
  - installer and uninstaller are created and usable.
