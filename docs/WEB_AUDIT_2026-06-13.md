## Performance

Major changes:

- Converted public routes to SSG/ISR where request-time data was unnecessary.
- Removed root `headers()` usage that made the entire public site dynamic.
- Split page-specific client components so home no longer includes Work, CV, Apps, Contact, or project-detail code.
- Converted CV and the MoPlayer hub to Server Components.
- Converted the global 404 page to a Server Component.
- Replaced automatic assistant loading with click-to-load.
- Reduced mobile backdrop blur and retained reduced-motion handling.
- Added accurate responsive image sizes and quality controls.
- Added 384px and 480px image breakpoints.
- Reduced the English and Arabic font critical path and limited the site to two families.
- Added `content-visibility` to long below-the-fold CV and MoPlayer sections.
- Replaced the oversized browser icon request with the 64px favicon, added a dedicated 180px Apple icon, corrected the PWA manifest icon dimensions, and switched social metadata to the generated 1200x630 Open Graph image.
- Recompressed the 512px transparent site icon from 67.7 KB to 22.1 KB without changing its dimensions or appearance.
- Added generated Open Graph and Twitter metadata routes to the proxy allowlist so their signed Next.js URLs are served directly instead of being locale-prefixed into 404 pages.

Modern-browser initial JavaScript for the main acceptance routes is approximately `157.5 KB gzip`, down from approximately `409 KB gzip` during the audit.

The score, accessibility, best-practices, SEO, JavaScript, CLS, and TBT budgets are met. The strict `LCP < 2.5s` target is not yet met under the local Lighthouse mobile profile. The remaining render delay is primarily the shared 48 KB global stylesheet. Reaching the strict LCP target safely requires splitting the legacy 10,000-line stylesheet into route-owned CSS rather than applying risky runtime CSS inlining; a measured `inlineCss` experiment increased HTML to about 1 MB and reduced performance, so it was removed.

Both public web and admin apps now use the current stable compatible release, Next.js `16.2.9` with `eslint-config-next` `16.2.9`.

`npm audit --omit=dev` still reports three transitive advisories:

- `esbuild` `0.27.7`, loaded by Vite/Vitest/tsx development tooling. The fixed `0.28.1` release is outside the current `tsx` compatibility range, so no unsafe lockfile override was added.
- A nested PostCSS `8.4.31` copy bundled under Next.js.
- Next.js is consequently listed as affected through that nested PostCSS copy. npm proposes downgrading to Next.js `9.3.3`, which is not a valid or safe remediation for this application.

These advisories should be rechecked when compatible upstream releases are available. They do not justify downgrading the framework or forcing an incompatible build-tool override.
